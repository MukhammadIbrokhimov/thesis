\chapter{Methodology}
\label{ch:methodology}

This chapter describes how I set up the experiments. I define the classification problem formally, explain how symptoms are represented as features, describe each of the five classifiers I chose and why, and then cover class imbalance handling, hyperparameter tuning, and the metrics I use for evaluation.

\section{Problem Formulation}
\label{sec:problem_formulation}

The task is multi-class supervised classification. Given a patient symptom vector $\mathbf{x} \in \{0, 1\}^d$, where $d$ is the number of distinct symptoms and each element indicates whether a symptom is present (1) or absent (0), the goal is to predict the disease label $y \in \{1, 2, \ldots, C\}$, where $C$ is the number of disease categories.

Formally, I want to learn a mapping $f: \{0, 1\}^d \rightarrow \{1, \ldots, C\}$ that minimizes classification error on unseen patients. I train the classifiers on a labeled training set $\mathcal{D} = \{(\mathbf{x}_i, y_i)\}_{i=1}^{N}$ and evaluate on a held-out test set.

This formulation makes three simplifying assumptions:
\begin{enumerate}
    \item The symptom representation is the same whether the symptom was self-reported or clinically observed.
    \item The datasets reasonably represent real symptom-disease patterns, though they're obviously simplified compared to actual clinical encounters.
    \item Each patient has exactly one disease label. In reality, patients often have multiple conditions simultaneously, but the datasets don't support multi-label classification.
\end{enumerate}

\section{Feature Representation}
\label{sec:feature_representation}

Each patient is represented as a binary vector of dimension $d = 132$ (for Dataset~1). For example, a patient with fever, headache, and nausea would have:

\begin{equation}
\label{eq:feature_vector}
\mathbf{x} = [\underbrace{0, \ldots, 0}_{\text{other symptoms}}, \underbrace{1}_{\text{fever}}, \underbrace{0, \ldots, 0}_{\text{other symptoms}}, \underbrace{1}_{\text{headache}}, \underbrace{0, \ldots, 0}_{\text{other symptoms}}, \underbrace{1}_{\text{nausea}}, \underbrace{0, \ldots, 0}_{\text{other symptoms}}],
\end{equation}

\noindent with 1s only at the positions of the reported symptoms and 0s everywhere else.

This is similar to the bag-of-words representation in text classification. It captures which symptoms are present but nothing about severity, duration, or order. That is a real limitation, but in practice, binary symptom vectors work surprisingly well for disease classification, especially with classifiers that can pick up on combinations of features.

\section{Classification Models}
\label{sec:models}

I selected five classifiers that span a range from simple to complex. The criteria were: (1) each method should have established use in medical classification, (2) the set should cover different algorithmic approaches (probabilistic, linear, kernel-based, ensemble), and (3) all should have mature implementations in scikit-learn or XGBoost.

\subsection{Naive Bayes}
\label{sec:naive_bayes}

I started with Naive Bayes as the simplest baseline. It applies Bayes' theorem assuming all features are conditionally independent given the class \citep{Bishop2006}. I use the Bernoulli variant since the features are binary:

\begin{equation}
\label{eq:naive_bayes}
P(y \mid \mathbf{x}) \propto P(y) \prod_{j=1}^{d} P(x_j \mid y).
\end{equation}

The independence assumption is clearly wrong --- fever and headache co-occur all the time. But I included Naive Bayes because it trains in under a second and gives a floor on what the more complex methods should beat. Implementation: \texttt{sklearn.naive\_bayes.BernoulliNB}.

\subsection{Logistic Regression}
\label{sec:logistic_regression}

Logistic Regression was my second choice, mainly for interpretability. It models class probabilities as a softmax over linear features \citep{Bishop2006}:

\begin{equation}
\label{eq:logistic_regression}
P(y = k \mid \mathbf{x}) = \frac{\exp(\mathbf{w}_k^T \mathbf{x} + b_k)}{\sum_{j=1}^{C} \exp(\mathbf{w}_j^T \mathbf{x} + b_j)}.
\end{equation}

I used L2 regularization to prevent overfitting. The reason I kept this one is that the learned weights $\mathbf{w}_k$ directly show which symptoms push toward which disease --- if a doctor asks ``why did it recommend a dermatologist?'', Logistic Regression can give a clear answer. Implementation: \texttt{sklearn.linear\_model.LogisticRegression} with \texttt{lbfgs}.

\subsection{Support Vector Machine}
\label{sec:svm}

SVM with an RBF kernel was included to test a non-linear method \citep{Cortes1995}:

\begin{equation}
\label{eq:svm_kernel}
K(\mathbf{x}_i, \mathbf{x}_j) = \exp\left(-\gamma \|\mathbf{x}_i - \mathbf{x}_j\|^2\right).
\end{equation}

For 41 classes, SVM trains $C(C-1)/2 = 820$ binary classifiers using one-vs-one. I quickly discovered this was painfully slow --- SVM took longer than all the other methods combined on Dataset~2. Implementation: \texttt{sklearn.svm.SVC}.

\subsection{Random Forest}
\label{sec:random_forest}

Random Forest was the first ensemble method I tested \citep{Breiman2001}. It builds many trees on bootstrap samples, each seeing only a random subset of features at each split:

\begin{equation}
\label{eq:random_forest}
\hat{y} = \text{mode}\{h_1(\mathbf{x}), h_2(\mathbf{x}), \ldots, h_T(\mathbf{x})\}.
\end{equation}

I chose this because it handles binary data well, needs no feature scaling, and gives feature importance scores (which I use later in Section~\ref{sec:feature_importance}). In early tests, 200 trees already gave solid results and trained in just a few seconds. Implementation: \texttt{sklearn.ensemble.RandomForestClassifier}.

\subsection{XGBoost}
\label{sec:xgboost}

XGBoost builds trees sequentially --- each new tree corrects the mistakes of previous ones \citep{Chen2016}:

\begin{equation}
\label{eq:xgboost}
\mathcal{L}^{(t)} = \sum_{i=1}^{N} l(y_i, \hat{y}_i^{(t-1)} + f_t(\mathbf{x}_i)) + \Omega(f_t).
\end{equation}

I included XGBoost mainly to see if its extra complexity over Random Forest actually buys anything for symptom data. It has a reputation for dominating Kaggle competitions on tabular datasets. The regularization term $\Omega$ should help with the smaller Dataset~1 where overfitting is more of a risk. Implementation: \texttt{xgboost.XGBClassifier}.

\section{Class Imbalance Handling}
\label{sec:class_imbalance}

Both datasets have class imbalance --- some diseases appear much more often than others. Without handling this, classifiers tend to over-predict common diseases and ignore rare ones.

I used two strategies:

\textbf{Class Weighting:} For Logistic Regression, SVM, Random Forest, and XGBoost, I set \texttt{class\_weight='balanced'}, which adjusts each class's weight inversely to its frequency:

\begin{equation}
\label{eq:class_weight}
w_k = \frac{N}{C \cdot n_k},
\end{equation}

\noindent where $N$ is the total number of samples, $C$ is the number of classes, and $n_k$ is the number of samples in class $k$. This means rare diseases get penalized more heavily in the loss function when misclassified.

\textbf{Stratified Sampling:} All train-test splits and cross-validation folds use stratified sampling to keep class proportions consistent.

I considered SMOTE \citep{Chawla2002} but rejected it. SMOTE generates synthetic examples by interpolating between existing ones, which doesn't work with binary features --- interpolating between two binary vectors produces fractional values that don't correspond to real symptom combinations \citep{Fernandez2018}. Class weighting achieves a similar effect without creating artificial data.

\section{Hyperparameter Tuning}
\label{sec:hyperparameter_tuning}

I tuned hyperparameters using grid search with stratified 5-fold cross-validation on the training set. The search spaces are in Table~\ref{tab:hyperparameters} in Appendix~\ref{app:supplementary}.

I used macro-averaged F1-score as the scoring metric for grid search instead of accuracy. The reason is that accuracy can be misleadingly high on imbalanced data --- a classifier that always predicts the most common disease could still get, say, 40\% accuracy. Macro-F1 treats every class equally, which forces the tuning process to care about rare-disease performance instead of just optimizing for the majority classes.

\section{Evaluation Metrics}
\label{sec:evaluation_metrics}

I evaluate each classifier using four metrics, all computed on the held-out test set.

\textbf{Accuracy} is the fraction of correct predictions:

\begin{equation}
\label{eq:accuracy}
\text{Accuracy} = \frac{\text{Number of correct predictions}}{\text{Total number of predictions}}.
\end{equation}

\textbf{Precision} measures how many of the predicted positives are actually correct for each class $k$:

\begin{equation}
\label{eq:precision}
\text{Precision}_k = \frac{TP_k}{TP_k + FP_k}.
\end{equation}

\textbf{Recall} measures how many of the actual positives are correctly identified:

\begin{equation}
\label{eq:recall}
\text{Recall}_k = \frac{TP_k}{TP_k + FN_k}.
\end{equation}

\textbf{F1-score} is the harmonic mean of precision and recall:

\begin{equation}
\label{eq:f1}
\text{F1}_k = 2 \cdot \frac{\text{Precision}_k \cdot \text{Recall}_k}{\text{Precision}_k + \text{Recall}_k}.
\end{equation}

I report macro-averaged versions of precision, recall, and F1-score:

\begin{equation}
\label{eq:macro}
\text{Macro-F1} = \frac{1}{C} \sum_{k=1}^{C} \text{F1}_k.
\end{equation}

I chose macro-averaging over micro-averaging because it treats every disease class equally. With micro-averaging, common diseases dominate the score and can hide poor performance on rare conditions. For a healthcare application, missing a rare but serious disease is worse than misclassifying a common cold, so equal weighting made more sense.

I also generated confusion matrices for the best-performing classifier to see which specific diseases get confused with each other.
