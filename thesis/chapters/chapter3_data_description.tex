\chapter{Data Description}
\label{ch:data_description}

This chapter describes the two publicly available datasets used in this thesis, their characteristics, and the preprocessing steps applied to prepare them for classification. Both datasets are sourced from Kaggle and represent symptom-based disease prediction tasks at different scales.

\section{Dataset 1: Disease Symptom Prediction Dataset}
\label{sec:dataset1}

The first dataset, the Disease Symptom Prediction Dataset \citep{Kaggle_Disease2020}, contains approximately 5,000 patient records. Each record maps a combination of symptoms to one of 41 disease categories. The dataset includes 132 unique symptoms represented as columns, where the presence of a symptom is indicated by a value of 1 and the absence by 0.

Table~\ref{tab:dataset1_overview} summarizes the key characteristics of this dataset.

The dataset was originally compiled from multiple medical sources and provides supplementary files including disease descriptions, symptom severity ratings, and suggested precautions. For the purposes of this thesis, only the symptom-disease mapping is used, as the primary objective is to evaluate classifier performance on the core prediction task.

The class distribution in Dataset~1 is moderately imbalanced. While most disease categories contain between 100 and 150 records, some rare diseases have fewer than 50 examples. This imbalance is addressed through the class-weighting and resampling techniques described in Section~\ref{sec:class_imbalance}.

\section{Dataset Comparison}
\label{sec:dataset_comparison}

The two datasets differ in several important respects beyond size. Dataset~1 is relatively clean but small, making it susceptible to overfitting, particularly for complex models with many parameters. Dataset~2 provides more data but may introduce additional noise due to its compilation methodology.

Using two datasets of different scales serves the analytical goal of this thesis: it allows the comparison of classifiers not only in terms of absolute performance but also in terms of how performance changes with data availability. This is particularly relevant for practical deployment, where the amount of available training data may vary significantly across healthcare contexts.

\section{Preprocessing Pipeline}
\label{sec:preprocessing}

A consistent preprocessing pipeline was applied to both datasets to ensure fair comparison across classifiers. The pipeline consists of the following steps:

\textbf{Step 1: Data Loading and Inspection.} Each dataset was loaded from its CSV format and inspected for structural integrity, including column types, missing values, and duplicate records.

\textbf{Step 2: Missing Value Handling.} For Dataset~1, missing values in symptom columns were imputed with 0 (symptom absent), consistent with the binary encoding scheme. Dataset~2 contained minimal missing values, which were handled identically.

\textbf{Step 3: Feature Encoding.} Symptoms were encoded as binary features, where each column represents a specific symptom. For Dataset~1, this encoding was already present in the raw data. For Dataset~2, text-based symptom entries were converted to the same binary vector representation to ensure consistency.

\textbf{Step 4: Label Encoding.} Disease names (the target variable) were encoded as integer labels using scikit-learn's \texttt{LabelEncoder}. The mapping between disease names and integer labels was preserved for interpretation of results.

\textbf{Step 5: Train-Test Split.} Each dataset was split into training (80\%) and testing (20\%) sets using stratified sampling to preserve the class distribution in both subsets.

\textbf{Step 6: Feature Scaling.} For classifiers sensitive to feature scale (Logistic Regression and SVM), features were standardized using \texttt{StandardScaler} to have zero mean and unit variance. Tree-based classifiers (Random Forest, XGBoost) and Naive Bayes do not require scaling and were trained on the original binary features.

\textbf{Step 7: Cross-Validation Setup.} Stratified 5-fold cross-validation was used for hyperparameter tuning and model evaluation. The stratified split ensures that each fold maintains the same class proportions as the full dataset, which is critical for imbalanced datasets.

The complete preprocessing pipeline, including software versions and hyperparameter search spaces, is documented in Appendix~\ref{app:supplementary}.
